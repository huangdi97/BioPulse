# ADR-0006：Vibe Coding 工作流

## 日期
2026-05-23

## 状态
已接受

## 背景
项目从零构建，没有专职开发人员。若衡是架构师/规划者，代码全部由 AI 编码助手生成。这种"设计 + AI 执行 + 人工验证"的模式需要一套可靠的工作流。

## 决策
**三阶段工作流：**

```
设计阶段 → 执行文档 + tasks.md → OpenCode → 审查 → 验收
```

- **设计阶段**：只给逻辑描述和架构约束，不给具体代码
- **文档先行**：每个批次的执行文档必须发 TG，详细到"新窗口拿到就能开工"
- **代码生成**：只通过 `opencode run --file tasks.md` 生成
- **审查**：若衡读取生成文件，对照 tasks.md 验收
- **两件套规则**：执行文档 + tasks.md 两份都必须发 TG 附件

## 理由
- **信息透明**：老板在任何时候都能看到"现在在做什么"
- **执行纪律**：跳过执行文档直接写代码 → 无法审查
- **断点续传**：有执行文档在，即使 OpenCode 出问题也能恢复

## 被否决的方案
- **直接写代码**：跳过设计文档，vibe coding → 无法追溯"为什么这么写"
- **只有 tasks.md 没有执行文档**：tasks.md 是给 OpenCode 看的，执行文档是给人看的

## 后果
- 每个开发批次多花 5-10 分钟写文档
- 文档与代码同步 —— 如果代码执行结果与文档不一致，先改文档
- 所有 18 条编码准则必须完整写入每个 tasks.md 和执行文档，不可简写
