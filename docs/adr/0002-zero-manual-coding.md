# ADR-0002：零容忍手动编码

## 日期
2026-05-23

## 状态
已接受

## 背景
项目采用 vibe coding 工作流（AI 生成 + 人工验证）。若衡（架构师角色）有权访问 `write_file` / `patch` / `terminal` 等工具，但若衡直接写代码会导致：
1. 代码风格不统一（AI 与手动混编）
2. 无法通过 OpenCode 的 lint 和格式检查
3. 维护时找不到"这段代码是谁写的"

## 决策
**任何人（包括若衡）不得手动写代码。** 所有代码必须通过 OpenCode CLI 生成。

若衡的职责：写 `tasks.md`（功能描述、验收标准）→ 调用 `opencode run --file tasks.md` → 审查 OpenCode 的输出 → 给出修改意见。

## 理由
- **单一入口**：所有代码由同一工具链生成，风格统一
- **审计可追溯**：代码变更全部通过 OpenCode，不走暗渠
- **纪律刚性**：一旦开手动修改的口子，"就改一行"会不断蔓延

## 被否决的方案
- **小改动手动修**：若衡认为"这种改一行就行"的场景直接修 → 纪律崩坏
- **delegate_task 写代码**：subagent 无法保证输出质量，且绕过 OpenCode 工作流
- **patch 修 bug**：测试导入路径等"小修" → 例外会逐步扩大

## 后果
- OpenCode 故障时不换工具，debug 原因后重试
- 若衡发现代码 bug 时：记录问题 → 写 tasks.md → opencode run → 验证结果
- 文件超 300 行时必须让 OpenCode 自己压缩，即使它当时卡住了
